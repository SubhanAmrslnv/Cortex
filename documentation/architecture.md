# Architecture

## System overview

Cortex is structured as three independent layers: a guard layer that fires before tool execution, a runtime layer that fires after, and a command layer that orchestrates analysis workflows. All layers read their configuration from a central JSON registry under `.cortex/registry/`. The adapter layer (`.claude/`) contains no logic — it only wires hook script paths to Claude Code events. The framework root (`CORTEX_ROOT`) is resolved dynamically at runtime so that Cortex can be installed globally, per-project, or via environment variable without any path hardcoding.

## Layer responsibilities

| Layer | Path | Responsibility |
|---|---|---|
| Guards | `core/hooks/guards/` | Gate logic only — block or inform before/after permission events |
| Runtime hooks | `core/hooks/runtime/` | Execution only — run after tools complete |
| Scanners | `core/scanners/` | Analysis only — called by `post-scan.sh` and `post-format.sh` |
| Commands | `commands/` | Orchestration only — no inline business logic |
| Registry | `registry/` | Configuration only — no executable code |
| Adapter | `.claude/` | Wiring only — no business logic |
| Cache | `cache/` | Generated state — read by prompt-optimizer, written by session-start |
| State | `state/` | Persistent snapshots — written by `/regression --save`, read by `/regression` |
| Local overrides | `local/` | Project customization — never modified by framework updates |

## Separation of concerns

- **Guards** may only block or enrich permission events. They must not write files or invoke scanners.
- **Runtime hooks** are read-only except for the audit log. They analyze and output structured JSON; they never modify the files they inspect.
- **Scanners** are called by `post-scan.sh` and `post-format.sh` via registry dispatch. They must not invoke other hooks.
- **Commands** read from the project and registry; they may invoke Bash tools and write reports, but they must not modify hook or registry files directly.
- **The adapter layer** (`.claude/`) contains only `settings.json` hook wiring and thin command wrappers. All logic lives in `.cortex/`.

## CORTEX_ROOT resolution

Every hook and command resolves its runtime path dynamically. Resolution order:

1. `$CORTEX_ROOT` environment variable — used in CI, Docker, and custom installs
2. `$(pwd)/.cortex` — project-local install detected at session start
3. `$HOME/.cortex` — global install fallback

Every hook script contains this block near the top:

```bash
if [ -z "$CORTEX_ROOT" ]; then
  if [ -d "$(pwd)/.cortex" ]; then
    export CORTEX_ROOT="$(pwd)/.cortex"
  else
    export CORTEX_ROOT="$HOME/.cortex"
  fi
fi
```

## Key design decisions

- **Registry-driven dispatch** — `post-scan.sh` and `post-format.sh` read `scanners.json` at runtime, so adding a new scanner requires only a registry entry and a script file; no hook changes needed
- **Numeric risk scoring** — `pre-guard.sh` accumulates points across 6 independent categories rather than pattern-matching against a blocklist; this allows partial matches to compound and intermediate-risk commands to warn rather than block
- **Idempotent session profiling** — `session-start.sh` fingerprints project files and skips rewriting the profile cache if nothing changed, keeping session startup fast
- **Snapshot-based regression** — `/regression` stores a fingerprinted snapshot of `/doctor` output and diffs against it, giving stable cross-session issue comparison without requiring external state
- **No hardcoded paths** — `settings.json` uses `${CORTEX_ROOT:-$HOME/.cortex}/...` for every hook path, making the adapter fully relocatable
